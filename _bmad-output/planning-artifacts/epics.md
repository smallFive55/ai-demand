# ai-demand - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for ai-demand, decomposing the requirements from the PRD, UX Design if it exists, and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: 业务方可通过对话方式提交需求，AI 支持多轮问答直至信息完整，过程可追溯。  
FR2: 系统 AI 根据需求内容自动识别所属项目，结果可展示并可人工修正。  
FR3: AI 根据业务板块准入标准判断需求是否符合准入，不符合时持续引导，直到符合或业务方确认放弃。  
FR4: 业务方可在对话过程中主动确认放弃需求，状态变更为“已放弃”，并保留对话记录。  
FR5: 需求接待成功后，通过企业微信通知对应业务板块的需求交付经理，包含需求概要与跳转链接。  
FR6: 需求被放弃时，通过企业微信通知需求提交者确认放弃。  
FR7: 需求接待成功后，AI 自动生成需求 PRD 文档，支持在线预览与下载。  
FR8: AI 自动生成 HTML 原型 DEMO，支持表单填写、按钮点击、页面跳转等至少 3 类交互。  
FR9: PRD 与原型生成完成后，企业微信通知需求交付经理审查。  
FR10: 需求交付经理可通过设计文档审查，通过后通知业务方评审。  
FR11: 需求交付经理可提交修改要求，系统记录修改意见并支持 AI 重新生成或人工介入。  
FR12: 需求交付经理可上传覆盖版本的 PRD 与原型，上传后覆盖 AI 版本并保留版本追溯。  
FR13: 交付经理通过后，企业微信通知需求提交者进行评审。  
FR14: 需求提交者可通过设计文档评审，通过后触发 AI 任务拆分。  
FR15: 需求提交者可提交修改要求，修改意见通知交付经理并进入后续处理。  
FR16: 需求提交者通过后，AI 自动进行任务拆分（概要设计、数据库、前端、后端、测试用例、API 自动化测试等）。  
FR17: 任务拆分完成后，企业微信通知需求交付经理审批任务拆分结果。  
FR18: 需求交付经理可审批任务拆分结果（通过/驳回）并记录审批意见。  
FR19: 任务审批通过后，AI 组建 AI 团队（架构师、前端、后端、测试）并分配明确任务。  
FR20: AI 架构师完成架构设计、数据库设计、接口设计并支持在线查看与下载。  
FR21: 架构设计完成后，企业微信通知需求交付经理进行架构审批。  
FR22: 架构审批通过后，其他 AI 成员可并行执行前端、后端、测试任务。  
FR23: 所有 AI 成员任务完成后，企业微信通知需求交付经理检查交付成果。  
FR24: 需求交付经理可审批交付成果（通过/需调整），需调整时记录要求并回流执行。  
FR25: 交付审批通过后，企业微信通知需求提交者进行验收。  
FR26: 需求提交者可验收需求（通过/不通过），不通过时可填写原因并退回交付经理。  
FR27: 需求验收通过 30 天后，企业微信通知需求交付经理跟进上线成果。  
FR28: 需求交付经理可填写/完善需求上线成果，支持文本、数据、附件。  
FR29: 上线成果完善后，系统自动生成需求复盘结果（需求概要、开发过程、上线成果、经验总结）。  
FR30: 复盘生成后，企业微信通知需求方与管理员并提供复盘报告链接。  
FR31: 管理员可创建业务板块，并配置板块名称、功能清单、需求交付经理、需求准入标准。  
FR32: 管理员可创建、编辑、禁用账号，支持账号与角色关联及批量导入。  
FR33: 管理员可定义角色并分配权限，实现权限与菜单、操作绑定。  
FR34: 管理员可配置菜单与操作级权限，支持按项目/业务线细粒度控制。  

### NonFunctional Requirements

NFR1: 系统在业务时间（8:00-22:00）可用性 >= 99.5%。  
NFR2: 页面加载性能 P95 <= 3 秒。  
NFR3: 非 AI 接口响应时间 P95 <= 500ms。  
NFR4: AI 任务（如 PRD 生成、任务拆分）必须支持异步执行并提供进度查询。  
NFR5: 所有状态变更与审批操作必须记录审计日志，日志保留 >= 1 年。  
NFR6: 支持企业微信扫码/SSO 登录，敏感操作要求二次确认。  
NFR7: 通知服务需与企微 API 解耦，支持失败重试与降级（站内信）。  
NFR8: AI 任务失败时必须通过企业微信通知需求交付经理，包含失败任务、原因摘要、建议人工介入动作。  

### Additional Requirements

- 架构明确要求首个实现故事执行项目脚手架初始化：前端采用 `create-vue`（Vue3 + Vite + TS + Router + Pinia + Vitest + ESLint + Prettier）。  
- 后端技术基线需覆盖 NestJS、MySQL、Redis、Node 22、REST/OpenAPI 与企业微信 OAuth2 SSO + JWT。  
- 需要事件驱动异步编排能力，支持 AI 任务、通知、审计等跨模块事件流转。  
- 需求状态流转需通过统一状态机服务处理，避免旁路更新导致不一致。  
- API 响应与错误结构需统一（success/error envelope），时间格式统一 ISO8601 UTC。  
- 数据库迁移要求版本化并纳入发布前强校验。  
- 关键外部集成包括企业微信 OAuth/消息 API 与大模型 API。  
- 前端需采用 features + shared 结构并按角色化路由与状态分层组织。  
- UX 要求 Web 为主平台，企业微信作为轻量通知与审批入口。  
- UX 要求“通知分层 + 一键操作”，审批通知需深链直达对应上下文。  
- UX 要求审批尽量单页完成，避免多跳链式审批。  
- UX 要求 AI 过程可视化：阶段进度、耗时反馈、完成双通道通知。  
- UX 要求异常提示采用“问题 + 原因 + 下一步”结构，并提供可恢复路径。  
- UX 明确响应式断点策略（sm/md/lg）与可访问性基线（WCAG 2.2 AA）。  
- 原型文件体现 UI 设计约束：左侧导航 + 顶栏 + 主内容区的工作台布局。  
- 原型文件体现核心流程界面：看板、对话提交、PRD/原型双视图、右侧审批面板、流转时间线。  
- 原型文件体现设计 token 与状态色语义映射（进行中/待审批/完成/失败）。

## Epic 1: 平台治理与权限基础

建立可用的组织治理底座（业务板块、账号、角色、权限），让后续需求接待与流转有清晰的责任归属和访问边界。

### Story 1.1: 从 Starter Template 初始化项目基线

As a 管理员,
I want 使用架构指定的 starter template 完成项目初始化与基础配置,
So that 团队可以在统一工程基线下继续实现后续治理与业务能力。

**Acceptance Criteria:**

**Given** 项目仓库已创建且尚未完成前后端工程初始化
**When** 团队按架构要求执行 starter template 初始化（含依赖安装与基础配置）
**Then** 项目生成可运行的基础工程结构并通过基础 lint/test/build 检查
**And** 工程中落地统一的目录约定、环境配置样例与脚本命令，支持后续故事开发

### Story 1.2: 管理员管理账号生命周期

As a 管理员,
I want 创建、编辑、禁用账号并支持批量导入,
So that 能高效维护组织成员与系统访问。

**Acceptance Criteria:**

**Given** 管理员已进入账号管理页面
**When** 管理员执行创建、编辑、禁用或批量导入操作
**Then** 账号状态与属性按操作结果更新
**And** 账号变更关联角色信息并保留操作审计记录

### Story 1.3: 管理员配置角色与权限矩阵

As a 管理员,
I want 定义角色并配置菜单和操作级权限,
So that 不同角色只能访问与操作其授权范围内的数据和功能。

**Acceptance Criteria:**

**Given** 管理员已进入角色权限配置页
**When** 管理员创建或编辑角色并分配菜单与操作级权限
**Then** 权限规则立即生效到对应角色账号
**And** 系统支持按项目或业务线进行细粒度权限配置与校验

### Story 1.4: 管理员配置业务板块准入规则

As a 管理员,
I want 创建并维护业务板块的功能清单、交付经理与准入标准,
So that 需求接待阶段可以正确识别板块并执行准入判断与路由。

**Acceptance Criteria:**

**Given** 管理员已完成基础权限配置并进入业务板块管理页
**When** 管理员新增或修改业务板块及其准入规则
**Then** 系统保存配置并在需求接待流程中可被实时读取与应用
**And** 修改记录纳入审计日志并支持后续追溯

## Epic 2: 对话式需求接待闭环

让业务方可以“说出需求即被系统受理”，并完成准入判断、放弃分支与关键通知。

### Story 2.1: 业务方对话式提交需求

As a 业务方,
I want 通过对话界面逐步描述需求并由 AI 引导补齐信息,
So that 无需复杂表单也能快速完成需求提交。

**Acceptance Criteria:**

**Given** 业务方已进入需求提交对话页面
**When** 业务方输入需求描述并继续多轮问答
**Then** 系统记录完整对话并持续收集需求关键字段
**And** 单轮 AI 响应满足时延目标且对话过程可追溯

### Story 2.2: AI 项目识别与准入判断

As a 业务方,
I want AI 自动识别所属项目并按准入标准判断是否可接待,
So that 我能知道需求是否具备进入后续流程的条件。

**Acceptance Criteria:**

**Given** 对话已收集到足够需求信息
**When** 系统执行项目识别与准入评估
**Then** 系统展示识别结果并允许人工修正
**And** 若不满足准入则给出补充引导，满足后将状态置为“已接待”

### Story 2.3: 放弃路径与通知闭环

As a 业务方,
I want 在不满足条件或不再推进时主动放弃需求,
So that 我可以结束本次流程且保留可追溯记录。

**Acceptance Criteria:**

**Given** 需求处于对话收集或准入引导阶段
**When** 业务方确认放弃
**Then** 系统将需求状态更新为“已放弃”并保留对话记录
**And** 系统通过企业微信通知提交者放弃结果

### Story 2.4: 接待成功通知交付经理

As a 需求交付经理,
I want 在需求接待成功后收到企业微信通知并可一键跳转详情,
So that 我能及时进入审查流程并减少等待。

**Acceptance Criteria:**

**Given** 某条需求已完成准入并进入“已接待”
**When** 系统触发接待成功通知
**Then** 对应业务板块交付经理在 1 分钟内收到企业微信消息
**And** 通知包含需求概要与可直达审批上下文的深链（至少携带 `requirementId`、`step`、`actionId` 参数）

## Epic 3: AI 方案生成与双轨评审

在接待后自动生成 PRD+原型，并完成交付经理与业务方双轨评审闭环。

### Story 3.1: 自动生成 PRD 与原型

As a 需求交付经理,
I want 系统在需求接待成功后自动生成 PRD 和交互原型,
So that 我能快速审查可执行方案而不是等待人工产出。

**Acceptance Criteria:**

**Given** 需求状态为“已接待”
**When** AI 生成流程完成
**Then** 系统产出可在线预览和下载的 PRD 文档
**And** 产出 HTML 原型且至少支持表单填写、按钮点击、页面跳转三类交互

### Story 3.2: 交付经理审查与版本控制

As a 需求交付经理,
I want 对 PRD/原型执行通过、提改或上传覆盖版本,
So that 可以在保证质量的前提下推进到业务评审。

**Acceptance Criteria:**

**Given** 交付经理已打开方案评审页
**When** 交付经理执行通过、提交修改要求或上传覆盖版本
**Then** 系统记录审查决策与意见并更新流程状态
**And** 上传覆盖版本后保留版本追溯关系与操作审计
**And** 审查流程在同一页面完成（内容预览 + 审批操作区），不依赖“列表->详情->表单”多跳链路

### Story 3.3: 业务方评审与回流机制

As a 业务方,
I want 在收到通知后评审 PRD/原型并选择通过或修改,
So that 方案能在业务目标对齐后再进入任务执行。

**Acceptance Criteria:**

**Given** 交付经理已通过审查并触发业务方评审
**When** 业务方执行通过或提交修改要求
**Then** 通过时流程进入任务拆分触发状态
**And** 修改意见回流至交付经理并保留完整评审记录
**And** 业务方从企业微信通知进入时，系统需基于深链参数自动定位到该需求与当前评审步骤

## Epic 4: 任务编排、执行与验收闭环

将评审通过的需求自动拆分任务、完成 AI 团队执行、交付审批与业务验收。

### Story 4.1: AI 任务拆分与审批

As a 需求交付经理,
I want 系统自动拆分任务并支持我审批拆分结果,
So that 我能确认执行计划符合交付预期。

**Acceptance Criteria:**

**Given** 业务方已通过方案评审
**When** 系统触发任务拆分并通知交付经理
**Then** 任务清单覆盖概要设计、数据库、前后端、测试和 API 自动化测试
**And** 交付经理可对拆分结果执行通过或驳回并记录审批意见
**And** 通知消息必须附带可直达当前审批步骤的深链参数（`requirementId`、`step`、`actionId`）

### Story 4.2: AI 团队组建与架构设计审批

As a 需求交付经理,
I want 在任务审批通过后查看 AI 团队分工与架构设计成果并审批,
So that 确保技术方案可执行且质量可控。

**Acceptance Criteria:**

**Given** 任务拆分已审批通过
**When** 系统组建 AI 团队并完成架构、数据库、接口设计
**Then** 交付经理可在线查看和下载相关设计产出
**And** 架构审批通过后系统允许后续并行执行前后端与测试任务
**And** 架构审批在同一页面完成查看与决策，避免多页面跳转审批

### Story 4.3: 交付成果检查与调整回流

As a 需求交付经理,
I want 在 AI 成员任务完成后审批交付成果并可要求调整,
So that 只有满足交付标准的结果才进入业务验收。

**Acceptance Criteria:**

**Given** AI 前后端与测试任务均已完成
**When** 交付经理检查交付成果并作出通过或需调整决策
**Then** 通过时系统通知业务方进入验收
**And** 需调整时系统记录调整要求并回流至执行阶段继续处理
**And** 交付经理从通知进入后应直接落到当前审批上下文并可单页完成审批

### Story 4.4: 业务验收与退回处理

As a 业务方,
I want 对交付成果进行通过或不通过验收,
So that 需求是否满足业务预期有明确结论并可追溯。

**Acceptance Criteria:**

**Given** 业务方已收到“待验收”通知
**When** 业务方提交验收结论
**Then** 通过时需求状态更新为“已验收”
**And** 不通过时必须填写原因并将流程退回交付经理处理
**And** 验收通知必须提供深链直达验收页面，且验收决策应在同页完成

## Epic 5: 上线跟进与复盘沉淀

在验收后持续跟进上线效果，自动生成复盘并通知相关角色，完成价值沉淀。

### Story 5.1: 验收后自动触发上线跟进

As a 需求交付经理,
I want 需求验收 30 天后自动收到上线成果跟进提醒,
So that 我能按时补齐业务效果并避免遗漏复盘。

**Acceptance Criteria:**

**Given** 某需求状态为“已验收”且达到 30 天
**When** 系统触发跟进提醒任务
**Then** 交付经理收到企业微信通知并可跳转填写入口
**And** 提醒记录纳入审计并支持重复通知防重

### Story 5.2: 上线成果录入与完善

As a 需求交付经理,
I want 填写上线结果并补充文本、数据和附件,
So that 复盘生成有完整且可验证的输入信息。

**Acceptance Criteria:**

**Given** 交付经理已进入上线成果编辑页
**When** 交付经理提交成果内容
**Then** 系统保存结构化成果数据并支持后续编辑
**And** 提交后触发复盘生成流程

### Story 5.3: 自动复盘生成与通知分发

As a 需求方或管理员,
I want 在成果完善后收到自动生成的复盘报告,
So that 可以复用经验并评估需求交付效果。

**Acceptance Criteria:**

**Given** 上线成果已完善且满足复盘生成条件
**When** 系统完成复盘报告生成
**Then** 报告包含需求概要、开发过程、上线成果与经验总结
**And** 系统通过企业微信向需求方与管理员发送可访问链接
