# 需求全流程管理系统 - 产品需求文档 (PRD)

**作者:** Peng  
**日期:** 2026-03-07  
**版本:** 1.2

---

## 一、执行摘要

### 1.0 问题陈述与市场机会

- **非 AI 时代局限**：市面项目协作管理系统多为非 AI 时代设计，缺乏 AI 深度参与，无法支撑从需求到交付的自动化闭环。
- **超级个体模式**：AI 时代下，单个需求交付者可完成接待、设计、开发、测试至上线全流程，需系统支持「一人全栈」的交付模式，而非传统产研多角色协作模式。

### 1.1 产品愿景

构建从**需求接待 → 需求分析 → 任务管理 → 需求总结**的全流程数字化管理系统，通过 AI 自动化与人工审批相结合，实现需求从提交到复盘的全生命周期可追溯、可管控。

### 1.2 核心差异化

- **AI 全流程参与**：对话式需求收集、需求识别、PRD 生成、原型 DEMO、任务拆分、团队组建、架构设计、数据库设计、接口设计、测试用例设计、API 自动化测试设计、代码生成均由 AI 完成
- **双轨审批机制**：需求交付经理与业务方分阶段审批，确保质量与业务对齐
- **企业微信深度集成**：关键节点自动通知，无需人工盯盘

### 1.3 目标用户


| 角色      | 描述        | 核心诉求            |
| ------- | --------- | --------------- |
| 业务方/需求方 | 提交需求、跟进进度 | 快速响应、进度透明、验收便捷  |
| 需求交付经理  | 全流程质量把控   | 审查效率、交付质量、成果可追溯 |
| 管理员     | 系统运维      | 账号、角色、权限统一管理    |

## 三、产品范围

### 3.1 MVP（最小可行产品）

- 三种角色：业务方、需求交付经理、管理员
- 需求接待：对话式需求收集、AI 项目识别、准入判断（符合或放弃）、企业微信通知
- 需求分析：AI 生成 PRD + HTML 原型、双轨审批（交付经理 → 业务方）
- 任务管理：AI 任务拆分、AI 团队组建、架构师设计、交付审批
- 需求总结：验收后 30 天提醒、复盘结果生成与通知
- 基础 RBAC：账号、角色、权限管理

### 3.2 增长阶段

- 多项目/多业务板块的交付经理分配规则配置
- 业务板块准入标准模板库与 AI 微调
- 需求模板与 PRD 模板库
- 数据看板与报表

### 3.3 愿景阶段

- AI 能力持续优化（大模型微调、RAG 增强）
- 与代码仓库、CI/CD、测试平台深度集成
- 需求价值度量与 ROI 分析

## 四、用户旅程

### 4.1 业务方旅程


| 阶段     | 动作             | 系统响应                              | 成功标准        |
| ------ | -------------- | --------------------------------- | ----------- |
| 提交需求   | 与 AI 对话式填写需求内容 | AI 识别项目、对话收集信息直至符合准入或业务方放弃、通知交付经理 | 收到接待成功/放弃反馈 |
| 等待设计   | 无主动操作          | 接收交付经理通知评审 PRD                    | -           |
| 评审 PRD | 通过 / 提交修改要求    | 通过则进入任务拆分；修改则交付经理处理               | 明确通过或修改路径   |
| 等待交付   | 无主动操作          | 接收验收通知                            | -           |
| 验收需求   | 通过 / 不通过       | 通过则进入总结阶段                         | 验收结果可追溯     |
| 接收复盘   | 查看复盘结果         | 企业微信推送复盘报告                        | 可访问完整复盘内容   |


### 4.2 需求交付经理旅程


| 阶段     | 动作               | 系统响应                  | 成功标准    |
| ------ | ---------------- | --------------------- | ------- |
| 接收接待通知 | 查看新需求            | 企业微信推送需求概要            | 可快速跳转系统 |
| 审查 PRD | 通过 / 修改要求 / 上传覆盖 | 通过则通知业务方；修改则 AI 或人工处理 | 审查意见可记录 |
| 审批任务拆分 | 通过 / 驳回          | 通过则 AI 组建团队           | 任务清单可调整 |
| 审批架构设计 | 通过 / 驳回          | 通过则 AI 成员执行开发         | 架构文档可下载 |
| 审批交付成果 | 通过 / 要求调整        | 通过则通知业务方验收            | 交付物可查看  |
| 跟进上线成果 | 填写上线成果           | 触发复盘生成                | 成果数据可编辑 |
| 接收复盘   | 查看复盘结果           | 企业微信推送                | 可导出复盘报告 |


### 4.3 管理员旅程


| 阶段     | 动作                           | 系统响应             | 成功标准       |
| ------ | ---------------------------- | ---------------- | ---------- |
| 业务板块管理 | 创建业务板块，配置板块名称、功能清单、交付经理、准入标准 | 业务板块与项目关联，支撑需求路由 | 板块可被需求识别引用 |
| 账号管理   | 创建/编辑/禁用账号                   | 账号与角色关联          | 支持批量导入     |

| 权限管理   | 配置菜单、操作权限                    | 细粒度控制            | 支持按项目/业务线  |

### 6.1 需求接待


| 编号      | 功能描述                                   | 验收标准                               | 追溯        |
| ------- | -------------------------------------- | ---------------------------------- | --------- |
| FR-R-01 | 业务方以**对话方式**提交需求，AI 自动与需求提交者对话完成需求信息收集 | 对话过程可追溯，支持多轮问答直至信息完整               | UJ-业务方-提交 |
| FR-R-02 | 系统 AI 根据需求内容自动识别所属项目（单个或多个）            | 识别结果可展示、可人工修正                      | SC-01     |
| FR-R-03 | AI 在对话中根据业务板块的准入标准判断需求是否符合准入条件         | 符合则进入「已接待」；不符合则继续引导补充，直至符合或业务方确认放弃 | SC-01     |

## 七、需求状态机

```
[业务方发起需求]
    → 对话收集中（AI 与业务方多轮对话）
    → (符合准入条件) 已接待
    → (业务方确认放弃) 已放弃

[已接待]
    → (AI 生成 PRD+原型) 待交付经理审查
    → (交付经理通过) 待业务方评审
    → (业务方通过) 待任务审批
    → (任务审批通过) 开发中
    → (架构审批通过) AI 执行中
    → (交付审批通过) 待业务方验收
    → (业务方验收通过) 已验收
    → (30 天后) 待跟进上线成果
    → (复盘完成) 已复盘
```

*注：各审批节点「驳回/修改」时，状态回退至对应上一阶段。*
