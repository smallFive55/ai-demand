# Story 2.3: 放弃路径与通知闭环

Status: in-progress

## Story

As a 业务方,
I want 在不满足条件或不再推进时主动放弃需求,
so that 我可以结束本次流程且保留可追溯记录。

## Acceptance Criteria

1. **Given** 需求处于 `collecting`（对话收集）或准入引导阶段，且业务方本人已登录 **When** 业务方在对话界面确认放弃 **Then** 系统将该需求状态更新为 `abandoned`（对应 PRD“已放弃”），并记录状态变更审计日志（包含 actor、from/to status、reason、timestamp）。
2. **Given** 业务方触发放弃操作 **When** 系统完成状态更新 **Then** 历史对话消息、字段快照与识别/准入结果保持可追溯，不得删除或覆盖；`GET /api/v1/requirements/:id` 仍可读取历史上下文用于追溯。
3. **Given** 需求已变为 `abandoned` **When** 任意用户继续通过对话接口追加消息 **Then** 返回统一错误信封（`success: false`），并给出“问题 + 原因 + 下一步”的可恢复提示，不允许继续走收集流程。
4. **Given** 需求进入 `abandoned` **When** 通知任务触发 **Then** 系统通过企业微信通知需求提交者放弃结果，通知内容至少包含需求标题/编号、放弃结论、查看详情深链（带 `requirementId`、`step`、`actionId`，可附 `source=wecom`）。
5. **Given** 企业微信通知发送失败或超时 **When** 异步任务执行 **Then** 系统执行失败重试并写入通知审计；超过重试阈值后降级为站内信/待办记录，满足 NFR7 的“解耦 + 重试 + 降级”要求。
6. **Given** 实现完成 **When** 执行测试 **Then** 至少覆盖：放弃权限校验、状态机迁移守卫、历史记录保留、企微通知参数契约、失败重试与降级路径。

## Tasks / Subtasks

- 任务 1：后端实现“放弃需求”状态迁移（AC: 1, 2, 3）
  - 在 `apps/api/src/modules/requirements/requirements.service.ts` 增加单点方法（如 `abandonRequirement`），统一封装 `collecting -> abandoned`，禁止控制器散落状态更新
  - 校验仅需求提交者本人可放弃，且仅允许 `collecting` 阶段（如覆盖“准入引导中”同一状态机分支，需在代码注释明确）
  - 扩展或复用审计动作类型，写入 `AuditService`
  - 保证放弃后历史消息与快照不被删除
- 任务 2：开放放弃接口并保持契约一致（AC: 1, 3）
  - 在 `requirements.controller.ts` 增加放弃 API（建议 `PATCH /api/v1/requirements/:id/abandon`）
  - 响应保持统一 `success/error` envelope 与 ISO8601 UTC 字段格式
  - 放弃后继续发消息时返回明确业务错误码与下一步提示
- 任务 3：通知闭环实现（AC: 4, 5）
  - 在通知模块或编排层新增“需求放弃”事件发布与消费（事件名遵循 `domain.entity.action.v1`）
  - 企业微信消息模板补充放弃通知，深链参数最小集合：`requirementId`、`step`、`actionId`
  - 增加失败重试与降级站内信逻辑，并记录通知链路审计
- 任务 4：前端交互与状态保护（AC: 1, 3, 4）
  - 在 `apps/web/src/views/requirement/RequirementChatView.vue` 增加“确认放弃”动作与二次确认弹层
  - 放弃成功后界面切换为只读，禁止继续发送消息，并展示状态说明
  - 若从企微深链进入，支持定位到对应需求并展示放弃结果上下文
- 任务 5：测试与回归（AC: 6）（后端部分）
  - Service/Controller 单测：权限、状态守卫、审计调用、错误信封
  - 通知模块测试：深链参数契约、重试/降级分支
  - 前端测试：放弃确认流程、放弃后禁发、错误提示文案结构

## Dev Notes

- 本故事聚焦 FR4 + FR6（放弃与通知），不包含接待成功通知交付经理（该项属于 Story 2.4）。
- 保持与 Story 2.2 已建立模式一致：单点状态迁移、统一错误信封、审计强制记录、可追溯数据保留。
- 通知链路需与企业微信 API 解耦，避免在业务事务内同步阻塞外部调用。

### Technical Requirements

- 后端基线：NestJS 11.x，Node 22 LTS，MySQL 8.4 LTS，Redis 8.x。
- 事件与状态：
  - 状态流转必须通过服务层单点方法；禁止控制器直接 `update(status)`
  - 事件命名：`domain.entity.action.v1`（例如 `requirement.status.abandoned.v1`）
- API 规范：
  - 统一响应信封：`{ success, data/meta }` / `{ success, error/meta }`
  - 日期时间统一 ISO8601 UTC

### Architecture Compliance

- 前后端模块映射遵循：
  - `web/features/intake` + `api/modules/requirements`（FR-R）
  - 通知相关通过 `api/modules/notifications` 或既有通知基础设施处理
- 审批/通知深链契约需保持与架构文档一致，避免仅跳列表页。

### Library / Framework Requirements

- NestJS 11 使用 Express v5 语义时，路由与中间件行为需按迁移指南核对。
- Node.js 22 作为 LTS 可继续沿用；如使用内置能力（如 WebSocket 客户端）需保持与现有封装一致，避免引入新依赖分叉。
- 企业微信消息接口需处理 token 生命周期、频率控制、错误码重试与降级。

### File Structure Requirements

- 预期重点修改路径：
  - `apps/api/src/modules/requirements/`
  - `apps/api/src/modules/notifications/`（或通知等价模块）
  - `apps/web/src/features/intake/`
  - `apps/web/src/views/requirement/RequirementChatView.vue`
- 不新增“平行实现”模块；优先复用 Story 2.1/2.2 既有服务与端口抽象。
