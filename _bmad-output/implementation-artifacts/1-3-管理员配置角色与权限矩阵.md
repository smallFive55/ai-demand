# Story 1.3: 管理员配置角色与权限矩阵

Status: ready-for-dev



## Story

As a 管理员,  
I want 定义角色并配置菜单和操作级权限,  
so that 不同角色只能访问与操作其授权范围内的数据和功能。

## Acceptance Criteria

1. **Given** 管理员已进入角色权限配置页
  **When** 管理员创建或编辑角色并分配菜单与操作级权限  
   **Then** 权限规则立即生效到对应角色账号  
   **And** 系统支持按项目或业务线进行细粒度权限配置与校验

## Tasks / Subtasks

- 建立角色与权限矩阵领域模型（AC: 1）
  - 在 `apps/api/src/modules/admin` 新增 `roles` 子模块（controller/service/types），支持角色创建、更新、启用/禁用、权限矩阵维护
  - 设计并实现菜单级与操作级权限结构（建议 `resource` + `actions[]` + `scope`），其中 `scope` 至少支持 `project` 与 `businessLine`
  - 将 Story 1.2 中 `AccountsService` 的硬编码 `validRoleIds` 替换为角色数据源查询，禁止继续依赖静态集合
- 打通权限生效与鉴权链路（AC: 1）
  - 在后端补充权限校验机制（Guard/Decorator），使角色矩阵真正作用于管理端接口访问
  - 定义并沉淀统一权限码命名规范（如 `admin.role.read`、`admin.role.manage`、`admin.account.disable`），避免后续故事重复命名
  - 增加权限变更审计事件（before/after、actor、target、requestId、occurredAt、success）
- 实现前端角色权限配置页（AC: 1）
  - 完成 `apps/web/src/views/admin/RoleView.vue`，从“暂无角色数据”升级为可用页面（角色列表、创建/编辑、权限勾选）
  - 在 `apps/web/src/features/admin` 新增角色管理能力（API 封装、类型定义、权限矩阵表单组件）
  - 提供项目/业务线维度的权限范围配置控件，保存前做前端校验并给出“问题 + 原因 + 下一步”反馈
- 落地契约与回归测试（AC: 1）
  - 在 `packages/contracts` 增加角色与权限 DTO，前后端复用，避免重复定义
  - 后端覆盖单测/集成测试：角色增改、权限矩阵保存、账号与角色联动、无权限访问拒绝、细粒度 scope 校验
  - 前端覆盖单测：角色创建编辑、权限勾选保存、范围切换、异常反馈；并执行 `pnpm lint`、`pnpm test`、`pnpm build`

## Dev Notes

### Story Foundation

- 本故事对应 FR33/FR34，是治理底座的核心控制面，直接决定后续流程是否能做到“按角色、按范围”安全访问。
- Story 1.2 已完成账号生命周期，但角色来源仍存在硬编码痕迹；本故事必须将“角色真实配置”升级为系统事实源。
- 验收重点不是“页面能保存”，而是“权限立即生效且可校验”，即接口层能真实阻断越权操作。

### Technical Requirements

- 后端维持 `NestJS 11 + Node 22` 基线，不引入与现有模块体系冲突的鉴权框架。
- 数据与接口格式保持现有规范：字段 `camelCase`、时间 ISO8601 UTC、响应 envelope `success/error + meta`。
- 权限矩阵需支持 `menu` 与 `action` 两层，不可只做菜单可见性；后端必须做最终权限判定，前端只做体验辅助。
- 角色更新后应立即影响账号行为，至少在下一次请求时生效，不允许“需重启服务才能生效”。

### Architecture Compliance

- 目录边界遵守既有约束：
  - 后端：`apps/api/src/modules/admin/`*（角色与权限领域），审计复用 `apps/api/src/modules/audit/*`
  - 前端：`apps/web/src/features/admin/*`（角色功能），通用复用放 `apps/web/src/shared/*`
- 状态与流程规则沿用统一服务层思想：权限判定逻辑不得散落在多个 controller 内。
- 与审批/通知深链契约保持兼容：本故事聚焦治理能力，不应改动业务审批路由参数协议（`requirementId`、`step`、`actionId`）。

### Library / Framework Requirements

- NestJS 11 最新维护分支可用（11.1.x），官方要求 Node >= 20；当前 Node 22 基线满足要求。
- MySQL 8.4 LTS 为长期支持线（适合生产长期演进）；新增角色权限模型时应避免与后续 MySQL 迁移冲突的临时结构。
- Redis 8 在 2025-2026 存在多次安全修复，若权限缓存落 Redis，需固定到最新稳定补丁并保留失效策略。

### Testing Requirements

- 后端必须覆盖：

  - `project/businessLine` 范围策略校验
- 前端必须覆盖：
  - 角色新增编辑流程
  - 菜单/操作权限勾选和回显
  - 范围切换（项目/业务线）与提交校验

- 质量门沿用：`pnpm lint`、`pnpm test`、`pnpm build` 全绿后进入 review。

### Previous Story Intelligence (Story 1.2)

- `AccountsService` 当前通过 `validRoleIds` 静态集合校验角色，属于临时实现；本故事必须替换为可维护的数据源。
- `AdminAuthGuard` 与 `x-request-id` 透传已落地，可直接复用，不应再造第二套鉴权入口。
- 审计服务已支持持久化与按 `requestId` 查询，权限变更审计应复用现有 `AuditService` 能力。
- 前端 `accounts` 能力已在 `apps/web/src/features/admin/accounts` 成型；角色模块可按相同组织模式实现，保持一致开发体验。

### Git Intelligence Summary

- 最近提交集中在“治理模块实装 + 可追溯增强”：`admin/accounts`、`audit`、`contracts` 是当前稳定扩展点。
- 已有代码风格倾向“先契约再实现 + 全链路 requestId + 审计优先”，本故事应延续该节奏。
- `RoleView` 仍为空壳，意味着本故事应以“补齐角色管理主路径”为主，不要分散到无关业务域改造。

### Latest Tech Information

- NestJS v11 迁移与入门文档明确 Node.js >= 20；当前项目 Node 22 基线可继续沿用。
- MySQL 8.4 LTS 属于稳定长期支持线，适合作为角色权限模型长期落库目标；8.0 已接近生命周期尾声。
- Redis 8 在近版本有安全修复记录，若引入权限缓存/会话联动，需强制补丁升级与安全基线检查。
