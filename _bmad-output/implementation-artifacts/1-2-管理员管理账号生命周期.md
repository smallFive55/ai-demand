# Story 1.2: 管理员管理账号生命周期

Status: ready-for-dev

## Story

As a 管理员,  
I want 创建、编辑、禁用账号并支持批量导入,  
so that 能高效维护组织成员与系统访问。

## Acceptance Criteria

1. **Given** 管理员已进入账号管理页面
  **When** 管理员执行创建、编辑、禁用或批量导入操作  
   **Then** 账号状态与属性按操作结果更新  
   **And** 账号变更关联角色信息并保留操作审计记录

## Tasks / Subtasks

- 建立账号管理领域模型与接口（AC: 1）
  - 在 `apps/api/src/modules/admin` 新增账户聚合（User/Account）与状态字段（启用/禁用）
  - 提供创建、编辑、禁用、批量导入 REST API，并统一返回 `success/error` envelope
  - 对角色关联做输入校验与存在性校验，禁止悬空 roleId
- 实现前端账号管理页关键流程（AC: 1）
  - 在 `apps/web/src/features/admin` 新增账号列表/编辑抽屉/禁用确认弹层/批量导入入口
  - 与 API 打通创建、编辑、禁用、导入流程，操作后刷新列表并展示结果反馈
  - 表单与批量导入错误提示采用“问题 + 原因 + 下一步”结构
- 落地审计与可追溯（AC: 1）
  - 为创建、编辑、禁用、导入写入审计事件（含 actor、target、before/after、时间）
  - 对失败操作保留失败审计（至少记录原因码与请求标识）
- 补充测试与质量门（AC: 1）
  - 后端：账户生命周期接口单测/集成测试
  - 前端：关键交互单测（表单提交、禁用确认、导入失败处理）
  - 回归执行 `pnpm lint`、`pnpm test`、`pnpm build`

## Dev Notes

### Story Foundation

- 本故事对应 FR32，是 Epic 1 治理能力的核心入口；后续角色权限矩阵（Story 1.3）依赖本故事账户与角色关联能力。
- 业务目标是“可维护组织成员访问生命周期”，不是一次性账号创建工具。
- 验收重点：状态变更准确、角色关联正确、审计完整可追溯。

### Technical Requirements

- 后端基线保持 NestJS v11 + Node 22，不得降级到旧运行时。
- 数据存储遵循架构文档（MySQL 8.4 LTS）；命名 `snake_case`、接口字段 `camelCase`。
- API 统一响应结构：成功 `{ success, data, meta }`，失败 `{ success, error, meta }`。
- 时间字段统一 ISO8601 UTC；审计事件至少包含 `occurredAt` 与 `requestId`。

### Architecture Compliance

- 目录边界必须遵守：
  - 后端：`apps/api/src/modules/admin`（领域逻辑）+ `apps/api/src/infra`（基础设施）
  - 前端：`apps/web/src/features/admin`（业务）+ `apps/web/src/shared`（复用）
- 禁止把账号管理逻辑散落到 `auth` 或无关模块，避免后续权限链路耦合失控。
- 账户状态变更必须经统一服务层，不得在 controller 直接写状态机逻辑。

### Library / Framework Requirements

- NestJS v11 与 Node 22 兼容（官方迁移指南要求 Node >=20）；当前基线可继续使用。
- Redis 8 已出现多次安全修复（8.0.x），如用于导入任务队列需锁定稳定补丁版本并及时更新。
- 企业微信 OAuth2 集成按授权码模式落地，回调 URI 必须白名单精确匹配，避免开放重定向风险。

### File Structure Requirements

- 后端建议新增：
  - `apps/api/src/modules/admin/accounts/`*（controller/service/dto/entities）
  - `apps/api/src/modules/audit/*`（若已存在则复用，不重复造轮子）
- 前端建议新增：
  - `apps/web/src/features/admin/accounts/*`（page/components/api/store）
- 共享契约优先落在 `packages/contracts`，避免前后端重复定义 DTO。

### Testing Requirements

- 后端必须覆盖：
  - 创建/编辑/禁用/导入成功路径
  - 非法 roleId、重复账号、导入格式错误等失败路径
  - 审计记录存在性断言
- 前端必须覆盖：
  - 账号创建编辑提交流程
  - 禁用二次确认
  - 批量导入错误反馈与部分成功场景
- 保持 Story 1.1 已建立的质量门：`lint/test/build` 全绿后再进入 review。

### Previous Story Intelligence (Story 1.1)

- 已形成 monorepo 与目录基线，不要新建第二套工程结构；在既有结构内增量实现。
- 已全局注册 API 响应拦截与异常过滤，新增接口应直接复用统一 envelope。
- CORS 已收敛到白名单配置，新增管理 API 无需重新放开全域访问。
- 鉴权 token 持久化链路已修复，前端请求应沿用现有 auth store 机制，不要旁路读写 token。
