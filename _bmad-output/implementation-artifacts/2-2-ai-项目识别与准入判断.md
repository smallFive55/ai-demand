# Story 2.2: AI 项目识别与准入判断

Status: done

## Story

**As a** 业务方,  
**I want** AI 自动识别所属项目并按准入标准判断是否可接待,  
**So that** 我能知道需求是否具备进入后续流程的条件。

## Acceptance Criteria

1. **Given** 某需求处于 `collecting` 且对话已收集到**足够信息**（与 Story 2.1 对齐：至少 `goalBackground`、`coreScope`、`successCriteria` 三类关键字段均已非空）**When** 系统在适当时机执行**项目识别 + 准入评估** **Then** 系统持久化**识别结果**（至少包含：拟归属业务板块 ID、匹配度分数或等价结构化结论、可选说明），并在前端**可展示**；业务方可通过专用 API/UI **人工修正**拟归属板块（修正后重新参与准入判断或直接进入下一状态，行为须在 Dev Notes 中单点定义并写测）
2. **Given** 已选定（AI 或人工）目标业务板块且该板块在库中存在 **When** 系统用该板块的 `admissionCriteria` 与 `admissionThreshold`（见 [Source: apps/api/src/database/entities/business-unit.entity.ts]）对当前需求摘要做准入判断 **Then** 若匹配度 **低于** 阈值，需求保持 `collecting`，AI 回复须体现**补充引导**（问题 + 原因 + 下一步，与架构错误结构一致）；若 **达到或高于** 阈值，将需求 `status` 更新为 `received`（与 PRD「已接待」及契约 `RequirementStatus` 一致），并写入审计
3. **Given** 需求已转为 `received` **When** 业务方再次尝试在对话中发送用户消息 **Then** 返回明确业务错误（或转入只读/完结态提示），避免在已接待后仍走「收集中」对话环（具体 HTTP 形态保持现有 `success/error` 信封）
4. **Given** 启用中的业务板块列表 **When** 业务方打开修正能力（如下拉选择）**Then** 可通过**业务方可访问**的只读 API 获取 `enabled` 板块列表（名称、id、功能清单摘要等），**不得**要求业务方调用管理员 `GET /admin/business-units/enabled`
5. **Given** 任意识别/准入/状态变更 **When** 操作完成 **Then** 关键动作写入 `AuditService`（与 2.1 一致），且非 AI 读写路径满足 NFR3（P95 ≤ 500ms）在合理数据量下的预期（单测或注释说明度量方式）

## Tasks / Subtasks

- **契约**（`packages/contracts/src/index.ts`）
  - 扩展接待相关类型：如 `AdmissionAssessment`、`Requirement` 上增加 `businessUnitId`、`projectIds`、`admissionScore`（或等价命名）等与 PRD「单个或多个项目」及展示/修正一致；与前端 `apps/web/src/types/requirement.ts`、Pinia **对齐**，以 contracts 为单一来源
- **数据库**（TypeORM + 迁移）
  - `requirements` 表增加：`business_unit_id`（可空）、`project_ids`（JSON 数组，默认 `[]`）、准入相关列（如 `admission_score`、`admission_rationale` 等按实现选用，须可验收 AC2）
  - 注册迁移脚本（沿用 `apps/api` 既有 migration 模式，参考 `1744291200000-CreateRequirementIntakeTables`）
- **板块只读 API**（`api/modules/requirements` 或独立轻量模块）
  - `GET /api/v1/business-units/enabled`（或嵌套在 `requirements` 下）：仅返回 `status === 'enabled'` 的板块；鉴权：`business`（及如需联调的 `admin`/`delivery_manager` 只读可沿用 2.1 读策略）
  - 复用 `BusinessUnitsService` 查询逻辑，避免在 requirements 中重复 SQL
- **识别与准入编排**（`RequirementsService` + LLM 抽象）
  - **选项 A（推荐）**：扩展 `LlmChatPort` / `HttpLlmChatService` 的 system prompt 与 JSON 尾行 schema，在字段齐备时输出 `suggestedBusinessUnitId`、`projectIds`、`admissionScore` 等（须校验 ID 属于启用板块，非法则降级为「无法识别」并引导）
  - **选项 B**：新增 `IntakeAssessmentPort`，在 `appendMessage` 末尾对「字段已齐」需求追加一次结构化调用（注意总耗时仍受 UX 10s 约束，必要时合并为单次模型调用）
  - 将 `collecting → received` 迁移抽成**单一私有方法**（便于未来替换为统一状态机服务），禁止在 Controller 多处直接改 `status`
  - 人工修正：`PATCH /api/v1/requirements/:id/intake`（示例）或等价 REST：仅允许业务方本人、且仅当 `collecting`，更新 `businessUnitId` 并可触发重算准入
- **前端**（`apps/web/src/features/intake/` + `RequirementChatView.vue`）
  - 展示识别结果卡片：板块名、项目 ID 列表、分数/结论；提供修正控件与保存
  - `status === 'received'` 时 UI 明确提示接待已完成，禁用发送或调用后端返回错误提示
- **测试**
  - Service：准入阈值上下界、`received` 后禁止发消息、非法 `businessUnitId`、审计调用
  - Controller：业务方只读板块列表、PATCH 权限

## Dev Notes

### 本故事范围边界（避免与 2.3、2.4 重叠）

- **本故事交付**：项目识别结果持久化与展示、人工修正、基于板块配置的准入判断、`collecting → received`。
