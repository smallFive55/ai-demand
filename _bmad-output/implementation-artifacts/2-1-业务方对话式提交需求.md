# Story 2.1: 业务方对话式提交需求

Status: done

## Story

**As a** 业务方,  
**I want** 通过对话界面逐步描述需求并由 AI 引导补齐信息,  
**So that** 无需复杂表单也能快速完成需求提交。

## Acceptance Criteria

1. **Given** 业务方（`role: business`）已登录并进入需求提交对话页（`/requirement/new`）**When** 用户发送自然语言描述并持续多轮输入 **Then** 服务端持久化完整对话消息序列（用户与 AI 交替、时间有序），前端刷新或重新进入同一会话仍可看到历史
2. **Given** 对话进行中 **When** 每轮用户发送消息 **Then** 系统在服务端更新「需求关键字段」结构化快照（至少覆盖 UX 与 PRD 对齐的三类信息：**目标/背景**、**核心功能或范围**、**预期效果或成功标准**；允许额外扩展字段但以这三类为验收最小集），且快照版本随对话递增可追溯
3. **Given** 用户发送消息 **When** AI 处理该轮并返回 **Then** 单轮从「请求进入后端」到「AI 回复内容可展示」的耗时在 **P95 ≤ 10 秒**（与 [Source: _bmad-output/planning-artifacts/ux-design-specification.md] 中单轮 AI 响应 ≤10s 一致）；若调用外部 LLM 超时，须返回符合架构要求的三段式错误（问题 + 原因 + 下一步），且不得丢失已持久化的用户消息
4. **Given** 任意对话消息已产生 **When** 监管或产品需要追溯 **Then** 可通过需求 ID 查询完整消息列表与字段快照历史（同一需求内按时间排序）；关键操作（创建需求、追加消息、更新快照）写入审计日志（可复用现有 `AuditService` 模式）
5. **Given** 非业务方角色（如 `admin` / `delivery_manager`）访问接待 API **When** 调用创建会话或发消息接口 **Then** 返回 403 或业务错误码，禁止冒充业务方提交（具体策略见 Dev Notes）

## Tasks / Subtasks

- **契约与类型**（`packages/contracts/src/index.ts`）
  - 定义 `Requirement`（接待阶段）与 `RequirementMessage`、`CollectedFields`（或等价命名）的 TypeScript 类型；与前端 `apps/web/src/types/requirement.ts`、Pinia `stores/requirement.ts` **对齐并消除重复定义**（以 `contracts` 为单一来源并在 web 中重导出）
- **数据库**（TypeORM 实体 + 迁移）
  - 新增 `requirements` 表：id、title（可占位）、status（默认 `collecting`）、submitterId、createdAt、updatedAt 等
  - 新增 `requirement_messages` 表：id、requirementId、role（user/ai/system）、content、createdAt
  - 新增「结构化收集」存储：可选用 JSON 列 `collected_fields` 于 requirements 表，或独立 `requirement_field_snapshots` 表（推荐独立表以便追溯「版本递增」）
  - 注册实体并纳入 `DatabaseModule`/`AppModule` 迁移流程（与现有 MySQL + TypeORM 模式一致，勿回退到 JSON 文件存储）
- **Backend — `api/modules/requirements`（或 `intake`）**
  - `RequirementsModule` / `RequirementsService` / `RequirementsController`
  - `POST /api/v1/requirements`：创建需求会话（status=`collecting`，submitter=`request.actor.id`）
  - `POST /api/v1/requirements/:id/messages`：追加用户消息 → 调用 AI 编排 → 持久化 AI 回复 → 更新 `collected fields` 快照
  - `GET /api/v1/requirements/:id/messages`：分页或全量拉取消息（供前端刷新恢复）
  - `GET /api/v1/requirements/:id`：详情（含当前最新字段快照）
  - 鉴权：仅 `role === 'business'` 可写；读权限与产品确认（默认：仅提交者本人可读自己的需求）
  - AI 调用：抽象 `LlmChatPort`（接口）+ 可切换实现（真实 HTTP LLM / 单测用 Fake）；禁止在 Controller 内直接写死第三方 SDK 调用逻辑
  - 审计：创建需求、消息落库、快照更新走 `AuditService`（与 admin 模块一致的成功/失败记录模式）
- **Auth / 角色**
  - 当前 `AuthService.login` 仅种子 admin；为本故事增加 **业务方种子账号**（`roleName: 'business'`）或文档化「联调 JWT」方案，确保 AC 中「业务方已登录」可验收
  - 评估将 `AdminAuthGuard` 重命名或抽取为通用 `BearerActorGuard`（可选，小步：若复用现有 Guard，须在 Dev Notes 中说明其虽名为 Admin 但实际解析任意角色 JWT/`admin:id` token）
- **Frontend — `apps/web/src/features/intake/`（与架构 FR-R 映射一致）**
  - 新建 `api.ts`：封装 requirements 相关 REST 调用（沿用 `apps/web/src/api/client.ts` 模式）
  - 重构 `RequirementChatView.vue`：进入页面时创建或恢复 `requirementId`（可用 sessionStorage 记住当前草稿 ID）；发送消息调用后端而非仅本地 `messages` 数组；展示 loading / 错误（`formatGuidedError` 若已存在则复用）
  - 更新 `stores/requirement.ts`：与后端列表/详情拉通（至少为本故事需要的部分）
  - UX：加载态、禁用重复发送、键盘 Enter 发送行为与现有页面一致；遵循 WCAG 对比度与 44px 触摸目标（见 UX 规范）
- **测试**
  - Service 单测：消息顺序、快照版本、权限拒绝、LLM 超时错误路径
  - Controller 单测（可 mock Guard）：契约与状态码
  - 前端关键交互单测（可选 Vitest）：发送后列表增加一条 user + 一条 ai（本迭代未新增 Vitest 用例；现有 web 测试已通过）

## Dev Notes

### 本故事范围边界（避免与 2.2–2.4 重叠）

- **本故事交付**：对话持久化、AI 多轮引导、关键字段结构化收集与追溯、性能与错误体验、业务方身份与 API 安全。
- **明确不在本故事**：项目/板块自动识别与人工修正（Story 2.2）、准入判断与「已接待」状态流转（2.2）、放弃流程与企微通知（2.3、2.6）、接待成功通知交付经理（2.4）。
- **状态机**：需求可保持 `collecting`；除非产品明确要求，本故事不要将状态推进到 `received`（进入 `received` 应留在 2.2）。

### 架构合规（必须遵守）

- **模块映射**：[Source: _bmad-output/planning-artifacts/architecture.md#Requirements-to-Structure-Mapping] — 需求接待（FR-R）→ `web/features/intake` + `api/modules/requirements`。
- **API 信封**：成功 `{ success, data, meta }`，失败 `{ success, error, meta }`；时间 ISO8601 UTC；字段 `camelCase`。[Source: architecture.md — Implementation Patterns]
- **错误提示**：对用户可见文案使用「问题 + 原因 + 下一步」。[Source: architecture.md — Process Patterns]
- **审计**：状态变更与关键操作必须可审计（NFR5）。[Source: epics.md — NFR5]
- **非 AI 接口性能**：普通 REST 读写 P95 ≤ 500ms（NFR3）；**AI 单轮**按 UX 10s 作为本故事 SLO，与 NFR3 区分对待。[Source: epics.md — NFR2/NFR3 + UX 规范]

### UX 约束（对话接待）
