# Story 1.4: 管理员配置业务板块准入规则

Status: done

## Story

**As a** 管理员,
**I want** 创建并维护业务板块的功能清单、交付经理与准入标准,
**So that** 需求接待阶段可以正确识别板块并执行准入判断与路由。

## Acceptance Criteria

1. **Given** 管理员已完成基础权限配置并进入业务板块管理页 **When** 管理员新增业务板块（含板块名称、功能清单、交付经理、准入标准与匹配度阈值）**Then** 系统保存配置并返回创建结果
2. **Given** 业务板块已创建 **When** 管理员修改板块信息或准入规则 **Then** 系统保存更新并在需求接待流程中可被实时读取与应用
3. **Given** 管理员执行新增或修改操作 **When** 操作成功或失败 **Then** 修改记录纳入审计日志并支持后续追溯
4. **Given** 管理员创建板块时选择交付经理 **When** 所选账号不存在或已禁用 **Then** 系统拒绝操作并给出三段式错误提示
5. **Given** 管理员输入板块名称 **When** 名称与已有板块重复 **Then** 系统拒绝操作并提示名称已存在
6. **Given** 板块列表中存在多条记录 **When** 管理员进入业务板块管理页 **Then** 系统展示板块列表（含名称、交付经理、状态），支持禁用/启用操作

## Dev Notes

### 架构模式与约束

- **严格遵循 accounts/roles 模块的实现模式**：相同的 Module/Controller/Service/Types 四件套、相同的 Guard 组合、相同的审计模式、相同的文件持久化模式
- **API 路径**：`/admin/business-units`（复数，kebab-case）
- **权限资源标识**：`admin.businessUnit`（camelCase，与现有 `admin.account`、`admin.role` 保持一致）
- **数据持久化**：JSON 文件存储（`.runtime-data/business-units.json`），不使用数据库（当前项目阶段统一方案）
- **审计 action 类型**：需在 `AuditService` 的 `AuditEvent.action` 联合类型中补充（如果 `create/update/disable/enable` 已覆盖则无需修改）

### 数据模型详细设计

```typescript
interface BusinessUnit {
  id: string                    // UUID
  name: string                  // 板块名称（唯一）
  description: string           // 板块描述
  functionList: string[]        // 功能清单（字符串数组，如 ["用户管理", "订单处理"]）
  deliveryManagerId: string     // 需求交付经理账号 ID（引用 AdminAccount.id）
  admissionCriteria: string     // 准入标准（自由文本，供 AI 读取判断）
  admissionThreshold: number    // 最低匹配度（0-100，默认 80）
  status: 'enabled' | 'disabled'
  createdAt: string             // ISO 8601 UTC
  updatedAt: string             // ISO 8601 UTC
}
```

**关键字段说明：**

- `functionList`：字符串数组，用于在需求接待时让 AI 判断需求属于哪个板块。PRD 原文："配置板块名称、板块功能清单"。前端实现为可动态增删的 Tag 输入组件。
- `admissionCriteria`：自由格式文本。PRD 待澄清项 1 已澄清："每个项目以文本格式录入准入标准，由 AI 自动判断需求与准入标准的匹配度"。该字段将在 Epic 2 Story 2.2 中被 AI 服务读取。
- `admissionThreshold`：整数 0-100。PRD："管理员可配置每个项目的最低匹配度要求（如：80% 匹配）"。默认值 80。
- `deliveryManagerId`：必须引用一个 status=enabled 的 AdminAccount。创建和更新时均需校验。

### 跨模块依赖

- `BusinessUnitsService` 依赖 `AccountsService`（校验 deliveryManagerId）和 `AuditService`（审计记录）
- `BusinessUnitsModule` 需导入 `AccountsModule`（AccountsModule 必须 exports AccountsService）和 `AuditModule`
- `AccountsModule` 当前是否已 export `AccountsService`？**需检查**——若未导出，需添加 `exports: [AccountsService]`

### 前端交互规范

- **Drawer/Dialog 模式**：完全复制 `RoleView.vue` 的交互模式（右侧抽屉、遮罩层、焦点捕获、Escape 关闭、遮罩点击关闭）
- **错误提示**：使用三段式 `formatGuidedError(问题, 原因, 下一步)` 函数，复制 RoleView.vue 中的实现
- **交付经理选择**：使用 `<select>` 或 Headless UI Listbox，数据来源为 `accountsApi.list()` 过滤 status=enabled 的账号
- **功能清单输入**：实现简单的 Tag 输入——输入框 + Enter 添加 + 每个 tag 有删除按钮；不需要使用第三方组件
- **准入标准输入**：`<textarea>` 多行文本，建议 rows=6
- **匹配度阈值**：`<input type="number" min="0" max="100" step="1">`，默认值 80

### UX 约束（来自 UX 设计规范）

- 颜色使用 Tailwind 主题 token（`text-text-primary`、`text-text-muted`、`bg-primary-600` 等），**禁止魔法色值**
- 所有可点击区域最小 44×44px 触摸目标
- 状态色语义：启用=success绿、禁用=muted灰
- 空状态：展示引导文案"暂无业务板块，点击上方按钮创建"
- Skeleton 加载态或简单 loading 文案

### 安全与校验

- 所有 Controller handler 必须使用 `@UseGuards(AdminAuthGuard, PermissionGuard)` + `@RequirePermission`
- Service 层校验：name 非空去重、deliveryManagerId 有效且启用、admissionThreshold 在 0-100 范围
- 失败操作也必须记录审计日志（try/catch 模式，参照 accounts/roles service）
