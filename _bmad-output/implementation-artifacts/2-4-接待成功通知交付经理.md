# Story 2.4: 接待成功通知交付经理

Status: done

## Story

As a 需求交付经理,
I want 在需求接待成功后收到企业微信通知并可一键跳转详情,
so that 我能及时进入审查流程并减少等待。

## Acceptance Criteria

1. **Given** 某需求在 `collecting` 阶段被判定为达到准入（LLM 自动路径 `llm_intake` 或业务方手动修正板块后的 `manual_patch` 路径） **When** `RequirementsService.applyCollectingToReceived` 完成状态迁移 **Then** 系统必须在同一事务流程中异步编排一次面向"该板块 `BusinessUnit.deliveryManagerId`"的企业微信通知（事件 `requirement.status.received.v1`），且通知调度不得阻塞 HTTP 响应。
2. **Given** 通知编排开始 **When** NotificationsService 向 `sendRequirementReceived` 发起首次调用 **Then** 系统先落 `notification_dispatch_started` 语义的 `requirement_received` 审计行（含 deepLinkParams / recipientTargeting / attemptedAt），随后进入重试循环；成功时再补一条成功审计，保持 outbox 语义一致性。
3. **Given** 企业微信正常响应（status=2xx 且 `errcode=0`） **When** 收到首次成功回包 **Then** 通知内容必须至少包含：需求标题、需求编号、提交者、拟归属板块名、匹配度分值、放行来源（llm_intake / manual_patch）、查看详情深链；深链路径默认 `/delivery-manager/approvals` 并附完整参数集（`requirementId`、`step=review`、`actionId`、`source=wecom`），绝对 URL 由 `APP_PUBLIC_URL`（回退 `APP_WEB_BASE_URL`）提供；路径允许通过 `NOTIFICATION_RECEIVED_DEEP_LINK_PATH` 环境变量覆写以对齐架构 `/{role}/approvals` 契约。
4. **Given** 企业微信发送失败（非 2xx / `errcode≠0` / timeout / URL 非法 / APP 基地址未配置） **When** NotificationsService 进入重试 **Then** 指数退避（基数 `NOTIFICATION_BACKOFF_BASE_MS` 默认 200ms，最小 10ms，上限 5_000ms）、最多 `NOTIFICATION_MAX_ATTEMPTS` 次（默认 3、范围 [1,8]）；重试耗尽后降级为站内信/待办（写入 `reasonCode=wecom_fallback_in_app_todo` 审计行），端到端总耗时在默认配置下可控制在 NFR-07 可接受区间内，并满足"1 分钟内"到达 SLA（可配置下调到极端不利时落降级也在该窗口内）。
5. **Given** 同一需求可能被并发触发 `applyCollectingToReceived`（例如 `manual_patch` 与 `llm_intake` 同时跑 race） **When** 状态已先被一方迁移到 `received` **Then** 后到者必须通过条件 UPDATE（`WHERE status='collecting'`）识别为 no-op，不得重复派发通知（"接待成功"应对应恰好一条 `requirement.status.received.v1` 事件）；no-op 路径写入 `reasonCode=received_concurrent_no_op` 审计行，幂等返回当前态。
6. **Given** `WECOM_WEBHOOK_URL` 未配置 **When** 系统进入接待成功路径 **Then** 选用 `NoopWecomNotifierService` 静默返回，不得抛错；仍必须写入 `dispatch_started` 与 `success=true, channel=wecom, externalMessageId=null` 审计行（用于开发/联调追踪），主流程对 noop 完全不可见。
7. **Given** 交付经理从企业微信深链进入 Web 前端 `/delivery-manager/approvals?requirementId=...&step=review&source=wecom` **When** 前端路由加载 **Then** 必须解析深链参数并定位目标需求（当前故事允许复用 `RequirementDetailView` 作为过渡落地页，后续独立审批页在 Story 3.2 / 4.1 切换）；参数无效或无权限时展示"问题 + 原因 + 下一步"三段式回退提示且不抛未捕获异常。
8. **Given** 实现完成 **When** 执行回归测试 **Then** 至少覆盖：通知端口放行契约（深链、mentioned_list、HTTPS 校验、APP_PUBLIC_URL 校验）；NotificationsService 成功/重试内成功/重试耗尽降级/dispatch_started outbox；`applyCollectingToReceived` 被 LLM 与手动路径共同触发时恰好 1 次通知调度；并发幂等；控制器 `PATCH /api/v1/requirements/:id/intake`（触发 `manual_patch` 分支）与 `POST /api/v1/requirements/:id/messages`（触发 `llm_intake` 分支）在通知调度被 mock 挂起 10s+ 的情形下 HTTP 响应 < 500ms（验证异步解耦）；现有 13 suites / 117 tests（Story 2.3 基线）不产生新回归。

## Dev Notes

- 本故事聚焦 FR5（接待成功通知交付经理），明确"单一事件对应单一通知"语义：`requirement.status.received.v1` 由 `applyCollectingToReceived` 单点触发；禁止从 controller 或其它 service 旁路调用通知（避免 Story 2.3 评审中指出的"审计乱序/重复派发"风险）。
- 深链契约遵循 `architecture.md:104`：`/{role}/approvals?requirementId=&step=&actionId=&source=wecom`；本故事 `role=delivery-manager`、`step=review`；环境变量 `NOTIFICATION_RECEIVED_DEEP_LINK_PATH` 仅供运维临时对齐历史路径使用，代码默认值即架构契约值。
- Story 2.3 已经把通知编排的"outbox 审计 → 重试 → 降级 → onApplicationShutdown"与"`track()` 在飞登记 + 优雅关闭排空"建成平台能力；本故事在 `NotificationsService` 内不得绕开这两条路径自定义实现，以保持可观测性与可运维性一致。
- Story 2.3 Round 3 留下的两个 `[AI-Review Round 3][FUTURE-STORY]`：① BullMQ/Redis Queue 工作队列替代进程内 fire-and-forget；② `<内部 userId → 企微 userid> 映射` 启用 `mentioned_list` 真实定向投递。这两项**不**作为本故事验收门禁，但在实现时必须：
  - 为后续 BullMQ 接入留出 Producer 语义（scheduleReceivedNotification 只调用 NotificationsService，绝不落业务 controller），避免日后再次大改；
  - 在载荷类型 `WecomRequirementReceivedPayload.mentionedWecomUserIds?` 预留字段并在 Http 实现中消费，以便未来拿到映射表时仅上层填充、下层不改。
- 1 分钟 SLA：NFR-07 + AC4 默认配置（3 次尝试 × 200ms 基数 × 5s HTTP 超时）最差约 ~15s；实际生产中"接待成功到企微消息到达"的 P95 期望 ≤ 10s。降级到站内信/待办的路径同样落在 1 分钟内（立即写审计行），不产生对 SLA 的"静默超时"风险。

### Technical Requirements

- 后端基线：NestJS 11.x、Node 22 LTS、MySQL 8.4 LTS、Redis 8.x（与 Story 2.3 一致）。
- 事件命名：`domain.entity.action.v1` → `requirement.status.received.v1`。
- 状态机单点：新增接待成功通知入口必须挂在 `RequirementsService.applyCollectingToReceived`，禁止在 `evaluateIntakeAndPersist` / `patchIntake` 里直接调用 `NotificationsService`（避免"两条路径各自派发一次"的重复面）。
- API 规范：本故事不新增 REST 路由；现有 `PATCH /api/v1/requirements/:id/intake` 与 `POST /api/v1/requirements/:id/messages` 的响应信封与错误三段式保持不变。
- ISO8601 UTC 时间格式在审计 `occurredAt` 与 `attemptedAt` 强制统一（与 Story 2.3 Round 2 H5 保持一致）。
